Getting Started with the SDK

Android library⚓︎

The Origo Mobile SDK for Android is provided as a single Android library (.aar) file. The SDK file comes in two variants:

  • The development SDK file has a -debug suffix and is intended to be used during development.
  • The production SDK file has a -release suffix and contains anti-tampering guards.

It is important to make sure you switch implementation before publishing your application.

Important

SDK cannot be downgraded once it has been upgraded to version "1.7.7" or higher. If we want to downgrade the SDK version, then it has to go as a fresh installation. Integrating apps should include the "androidx.security:security-crypto:1.1.0-alpha03" API as transitive dependencies starting with SDK version "1.7.7" in their application build.gradle.

Back Up User Data/Auto Backup⚓︎

The HID Origo Android SDK stores encrypted, non-transferable data on the device. Therefore, using the Auto Backup functionality provided by default in Android 6.0 (API level 23) and later is not supported. The easiest way to disable Auto Backup is by inserting the following in your AndroidManifest.xml file.

1
2
3
4
5
6
7
8
9
<manifest ... >
    ...
    <application
        tools:replace="android:fullBackupContent"
         android:allowBackup="false"
         android:fullBackupContent = "true" ... >
        ...
    </application>
</manifest>

If your application targets Android 6.0 (API level 23) android:allowBackup will default to true, so you will need to set the above explicitly.

Enabling Auto Backup⚓︎

If you need to enable Auto Backup for your application you need to make certain adjustments to your app configuration. As of Origo SDK 1.3.0 this is done automatically by the SDK, but if you are using an earlier version, or if you need to exclude other files from being backed up, you need to manually specify all of them.

This is done by following the instructions on https://developer.android.com/guide/topics/data/autobackup.html#IncludingFiles

Your fullBackupContent resource must exclude the following files:

A note on different phone models⚓︎

The measured RSSI Values could differ between different phone model, BLE chipsets, phone antennas, and phone firmwares.

On certain phones, NFC is configured to use the SIM card only. One example is certain Samsung phones after upgrading them to Android 5.x. On these phones, the NFC processing method needs to be changed to "Android" instead of SIM for HCE to work.

Origo Mobile keys API⚓︎

The main class in the SDK is com.hid.origo.api.OrigoMobileKeysApi. This class provides a centralized entry point for the Seos mobile integration.

OrigoMobileKeysApi contains three parts:

  • com.hid.origo.api.OrigoMobileKeys. Responsible for both Seos TSM communication and Seos secure element functionality.
  • com.hid.origo.api.OrigoReaderConnectionController. Used for controlling the Bluetooth and NFC reader communication.
  • com.hid.origo.api.OrigoDeviceEligibility. Contains device specific configurations and information about working/non-working features.

Initializing the API⚓︎

The OrigoMobileKeysApi interface is obtained using the .getInstance() method, however to access the OrigoMobileKeys and OrigoReaderConnectionController interfaces, the API must be initialized.

Firstly, make sure that the API is not already initialized by calling .isInitialized(). Then initialize the API using the following method:

.initialize (Context context, OrigoApiConfiguration origoApiConfiguration, OrigoScanConfiguration origoScanConfiguration, String applicationId).

Origo Favorite Reader⚓︎

To integrate the favorite reader in your application, you need to create the instance for com.hid.origo.api.favoriteReader.OrigoFavoriteReaderManager. This provides methods for you to add, remove and retrieve favorite readers. Please refer to Javadoc for more technical details.

Initializing favorite reader⚓︎

The OrigoFavoriteReaderManager is a singleton class and you need to initialize the class in your application and need to call getInstance() method it will return the instance of OrigoFavoriteReaderManager.

Add favorite reader⚓︎

To add favorite reader, you need to call addFavoriteReader(@NonNull OrigoFavoriteReader reader) method in your application. Make sure OrigoFavoriteReader NonNull parameters assigned properly. OrigoFavoriteReader in this class address is the mandatory parameter. Please refer to the Error messages section for error details.

Remove favorite reader⚓︎

To remove the favorite reader, you need to call removeFavoriteReader(@NonNull OrigoFavoriteReader reader) method in your application and you need to pass the OrigoFavoriteReader object as a parameter. Please refer to the Error messages section for error details.

Remove all favorite reader⚓︎

To remove all favorite reader, you need to call removeAllFavoriteReaders() method in your application.

Get all favorite readers⚓︎

You need to call getFavoriteReaders() method in your application and this method will return the list of favorite readers.

Get nearby readers based on priority⚓︎

You need to call getPrioritizedNearByReaders() method in your application and this method will return nearby readers which are sorted by favorite readers at the top and strong RSSI(Received signal strength indication) signal readers will be the next.

Update favorite reader⚓︎

To update a specific favorite reader, you need to call updateFavoriteReader(@NonNull OrigoFavoriteReader origoFavoriteReader) method in your application and user need to pass the OrigoFavoriteReader object as a parameter. Make sure the OrigoFavoriteReader NonNull parameter is assigned properly. OrigoFavoriteReader in this class address is the mandatory parameter and readerName, readerLastSeen, readerPSN can be updated except address. Please refer to the Error messages section for error details.

Error messages⚓︎

While implementing favorite reader in your application, you may get following Error messages :

Add favorite reader⚓︎

  • Mandatory fields empty: If address field is empty, this method will throw an exception Reader address missing, not eligible to add as a Favorite reader.
  • Duplicate Favorite Reader: If favorite reader is already added, this method will throw an exception Reader already added as a favorite reader.

Remove favorite reader⚓︎

  • Reader not available: If the given reader is not added as a favorite reader, Then If you try to remove it from the favorite list, this method will throw an exception Reader is not available in the favorite readers.

Update favorite reader⚓︎

  • Reader not available: If the given reader is not added as a favorite reader, then If you try to update it in the favorite list, this method will throw an exception Reader is not available in the favorite readers.

Origo SDK BLE Calibration⚓︎

The BLE calibration feature(only for TAP gesture) in the Origo SDK allows mobile devices to fine-tune the read range with readers. This is specifically useful for applications where users want control over how close the mobile device needs to be to a reader for opening via BLE Tap gesture.

This feature is provided through the OrigoReaderCalibrationManager class, which lets developers calibrate both specific and multiple readers at different distances, making it flexible for various use cases.

Why Use BLE Calibration?⚓︎

This BLE calibration feature addresses the challenge of varying read distances in environments where precise control over user proximity is needed. With this, applications can deliver a more reliable, personalized experience, ensuring users interact with readers at their preferred distances.

Many Android devices have BLE signals of varying strengths, causing them to communicate at different distances. BLE calibration helps standardize these interactions across diverse devices, allowing for more consistent performance and user experience.

Does the BLE sensitivity option still have support?⚓︎

Yes, BLE sensitivity is still supported.

Key Concepts⚓︎

Calibration Modes and Read Ranges:⚓︎

To utilize the BLE calibration feature within the OrigoReaderCalibrationManager, developers need to first create a Calibration object. This object encapsulates the calibration settings, such as the reader's address, the desired read range, and whether the calibration should apply to all readers or a specific one. The Calibration object is then used when calling methods in OrigoReaderCalibrationManager to manage calibration data (i.e., to add, update, or remove calibration).

There are two primary ways to instantiate a Calibration object, depending on the type of calibration desired:

1. Custom Range Calibration:⚓︎

For precise custom calibration, developers can specify a customRangeOffset along with ReadRange.CUSTOM. This is ideal for cases where the default ranges (e.g., NEARBY, FAR) don’t meet the application’s specific needs. The range can be changed within 10dBm only to avoid unexpected behaviour.

1
2
3
4
5
6
val calibration = Calibration(
readerAddress = "reader-123",
readRange = OrigoReaderCalibrationManager.ReadRange.CUSTOM,
customRange = 7,
isApplyToAllReaders = false
)

2. Predefined Range Calibration:⚓︎

For scenarios where one of the predefined ranges (e.g., NEARBY, FAR) is sufficient, developers can directly set a ReadRange value without specifying a custom offset.

1
2
3
4
5
val calibration = Calibration(
readerAddress = "reader-123",
readRange = OrigoReaderCalibrationManager.ReadRange.NEARBY,
isApplyToAllReaders = true
)

The ReadRange enum defines several preset distances:

Enum Value (dBm)
TOO_NEARBY -10
NEARBY -5
CUSTOM 0
FAR 5
TOO_FAR 10

Minimum & Maximum Calibration Range⚓︎

Pre-defined options Custom (dBm)
Minimum TOO_NEARBY -10
Maximum TOO_FAR 10

After creating the Calibration object, it can be passed to OrigoReaderCalibrationManager methods like addCalibration(), updateCalibration(), and removeCalibration() to manage the calibration data for the specified reader(s).

Exclusive Calibration:⚓︎

Applies to specific readers. This is useful for targeting individual readers with tailored settings.

Common Calibration:⚓︎

Applies globally to all readers unless overridden by an exclusive calibration.

Important Methods⚓︎

addCalibration(calibration: Calibration):⚓︎

Adds a new calibration. If isApplyToAllReaders is true, it’s stored as a common calibration; otherwise, it’s saved as exclusive for a specific reader.

removeCalibration(calibration: Calibration):⚓︎

Removes an existing calibration for a specific reader(exclusively calibrated). Common calibration is removed separately using removeCommonReaderCalibration().

updateCalibration(calibration: Calibration):⚓︎

Updates an existing calibration. Allows adjustment of the read range or other parameters as needed.

getCalibrationIfPresent(readerAddress: String):⚓︎

Fetches calibration data for a specific reader, if available.

getExclusiveCalibrations()⚓︎

Retrieve lists of exclusive calibrations.

getCommonReadersCalibration():⚓︎

Retrieve lists of common calibrations.

removeCommonReaderCalibration():⚓︎

Remove common calibration.

removeAllExclusiveCalibrations():⚓︎

Remove all lists of exclusive calibrations.

What is RASP (runtime application self-protection)?⚓︎

RASP (runtime application self-protection) security enables apps to monitor for suspicious behavior at runtime. When a runtime threat is detected, the RASP features help defend against threat actors attempting to tamper with your app or perform a dynamic analysis.

Hook Detection⚓︎

  • Identifies if runtime hooking frameworks are being used to intercept or modify app behavior.
  • Risk: Attackers can manipulate app logic or bypass security checks.
  • Defense: The SDK will crash the app if Hooking detection is triggered..

Debug Detection⚓︎

  • Detects if a debugger is attached to the app during runtime.

  • Risk: Debuggers can be used to step through code, alter variables, or expose sensitive data.

  • Defense: The SDK will crash the app if debug detection is triggered..

Emulator Detection⚓︎

  • Identifies if the release signed app is running on an emulator instead of a physical device.

  • Risk: Emulators facilitate reverse engineering and automated attacks in a controlled environment.

  • Defense: The SDK will crash the app if emulator detection is triggered.

Root Detection⚓︎

  • Checks if the device is rooted or jailbroken, giving users elevated permissions.

  • Risk: Root access can bypass app security, modify files, or inject malicious code.

  • Defense: Root detection will not crash the app. This is logged in the portal for jailbroken or rooted devices so the admin can take further action.

Note

The release SDK will not run in debug build. For example, if the release AAR is integrated and the app is running as a debug build, the SDK will crash the app.

Seos TSM integration⚓︎

There are five methods in the API to handle Seos TSM integration. These are located in the com.hid.origo.api.OrigoMobileKeys interface.

public interface OrigoMobileKeys
{
    void applicationStartup(OrigoMobileKeysCallback mobileKeysCallback, OrigoApplicationProperty... applicationProperties);

    void endpointSetup(OrigoMobileKeysCallback mobileKeysCallback, String activationCode, OrigoApplicationProperty... applicationProperties);

    void endpointUpdate(OrigoMobileKeysProgressCallback mobileKeysProgressCallback);

    boolean isEndpointSetupComplete() throws OrigoMobileKeysException;

    void unregisterEndpoint(OrigoMobileKeysProgressCallback mobileKeysCallback);
    //...
}

applicationStartup(...), endpointUpdate(...), and endpointSetup(...) are time consuming operations. They are executed asynchronously on a background thread and the result is delivered to the calling application, as the first parameter, via the com.hid.origo.api.OrigoMobileKeysCallback instance.

com.hid.origo.api.OrigoMobileKeysProgressCallbackis a subclass of the callback interface and has an additional method of sending progress events during a setup or update. However, as there is no way of telling how many events will occur before the setup/update is finished, this information cannot be used for filling a progress bar, although it could be used for giving the user some other form of feedback.

The call that takes the most time is the endpoint setup. This process could take anywhere between a few seconds and up to about a minute. Therefore, it is important to set user expectations before the process is started and provide feedback during the process, otherwise the user may assume the application is not responding.

Application startup⚓︎

The application startup method should be called during the startup phase of your application. This method ensures that the mobile device is in sync with the Seos TSM server and:

  • Sends properties about the installation to the Seos TSM, for example, version of the SDK or Android OS. The SDK detects when a property has changed and only performs an update if necessary.
  • Detects interrupted communication with the Seos TSM and attempts to resume communication.

Under normal circumstances server communication should not be interrupted. If, however, there is an application crash or memory issues on the mobile device, and the application is shut down, the SDK continues with outgoing messages to the Seos TSM. If, during application startup, there is a message pending, it is sent to the Seos TSM. This process may take some time, since the message might trigger a response from the server requiring more actions on the mobile device side and so on.

This is generally a fast operation as it does not communication with the Seos TSM, however it may require more time if Seos TSM interaction is needed. The calling application needs to handle the fact that this call might take some time before it is finished.

Endpoint setup⚓︎

An endpoint (a Seos instance in a secure element), must be installed and personalized before it can be used to open locks. The application firstly needs to check if the endpoint setup on the device is completed using the following method: com.hid.origo.api.OrigoMobileKeys.isEndpointSetupComplete

If the endpoint setup on the device is not complete, the application needs to call the following and wait for the process to finish:

com.hid.origo.api.OrigoMobileKeys.endpointSetup(OrigoMobileKeysCallback mobileKeysCallback, String activationCode, OrigoApplicationProperty... applicationProperties)

As part of the registration process the end user is provided with an invitation code. This code is used to identify the endpoint in the Seos TSM and to secure the setup process. The code is four-times-four alphanumerical characters separated by dashes (XXXX-XXXX-XXXX-XXXX). It is a required parameter to the endpoint setup method.

If the user is already authenticated, for example, using a backend server outside the TSM environment, there is no need to send the code to the user. The application can retrieve the invitation code from the backend server and feed it directly to the setup method in the SDK. In this way the user is transparently enrolled.

Note, the endpoint setup may fail on a mobile data network as it includes multiple roundtrips between the SDK and the Seos TSM. Therefore, when the mobile device has a stable network connection, the application can re-try calling the endpoint setup using the same invitation code. The only exception is if the SDK reports an error with the following code:

com.hid.origo.api.OrigoMobileKeysApiErrorCode.DEVICE_SETUP_FAILED.

This means that the setup failed in a critical phase, and for security reasons, the application must restart using with a new invitation code.

Combining SEOS, Wallet, OIP⚓︎

  • Existing endpointSetup integration:

    For integrators who wants to use only SEOS, they can continue using the OrigoMobileKeysCallback interface. Use the following method for endpoint setup:

Origo Organization specific contact information⚓︎

We would provide metadata to the APP with contact info so that the users would know whom to contact in case of problems.

To integrate the Organization specific contact information in your application, The getAdditionalMetadata() of OrigoMobileKey provides additional metadata that contain the methods to fetch Organization specific contact information fields.

Content updates⚓︎

Updates such as new keys, revocation of keys, etc. are delivered to the mobile device from the Seos TSM. To download and process these updates, the application needs to call the following:

com.hid.origo.api.OrigoMobileKeys.endpointUpdate(OrigoMobileKeysProgressCallback mobileKeysProgressCallback)

The application can update content every time it is launched, or the application backend can use a push mechanism to tell the application when an update is available.

Talking to Seos in the secure element⚓︎

For the most common use cases, the com.hid.origo.api.OrigoMobileKeys interface contains functionality to communicate with Seos. The more advanced use cases can use the Seos Access API. This provides a high-level Java API for integrating with the Seos Card Technology. For detailed information on how to use it, refer to the Seos Access API documentation.

Due to the single threaded nature of most secure elements, the OrigoMobileKeys implementation ensures that only one task is performed at a time. For example, one current request to list keys or a one-time consuming Seos TSM request. If there is already a request processing, or a Seos TSM task running, a com.hid.origo.api.OrigoMobileKeysException with the error code API_IS_BUSY will occur.

Listing issued keys⚓︎

To retrieve a list of MobileKey objects (representing all keys issued to this secure element), use the listMobileKeys() method. Each mobile key holds metadata about the key, for example, label, valid from/to dates, etc. If an application needs more information about the key, the metadata field configuration URL can be used to fetch more information from a remote server. For example, in our reference application we point this configuration URL to an XML document describing key card layout, background image, font size, color etc.

Reading metadata about Seos⚓︎

The getEndpointInfo() method provides metadata about the Seos installation. See com.hid.origo.api.OrigoEndpointInfo for more details about available data. This data is typically not useful for an end user but could be helpful for a support technician. We therefore encourage integrators to make this information available in some kind of help, support or about screen.

Observing changes⚓︎

It is possible to listen to keys and endpoint changes in the SDK by adding listeners to the com.hid.origo.api.OrigoMobileKeys interface.

Bluetooth reader integration⚓︎

The mobile device communicates with Bluetooth readers using Bluetooth Low Energy (also known as Bluetooth LE, BLE or Bluetooth Smart). Bluetooth LE was released as part of the Bluetooth 4.0 Core Specification, and Android support was first released as part of Android 4.3.

The Android 5.0 release significantly improved the Bluetooth LE support in Android, for example, less power consumption, and support for scanning with the screen off. With the increased user integrity and privacy of Android 6.0, Bluetooth LE scanning now requires location permissions.

Bluetooth implementation is hidden behind the following interface: com.hid.origo.api.OrigoReaderConnectionController

As an integrator user need to verify that the application supports Bluetooth before scanning. For this purpose, user can use the OrigoBleSupportHelper class in the in SDK. This helps to check if a device has the features required to be used with Bluetooth readers.

BLE configuration⚓︎

When the scan process is started in the BLE service you need to provide a OrigoScanConfiguration instance. The scan configuration contains the following parameters.

Parameter Description
Lock service code(s) Bluetooth readers advertise a UUID (universally unique identifier) identifying this BLE device as a reader.
The lock service code is part of that UUID. The UUID might differ between different types of readers.
Consult with your lock provider to receive the service code used by the reader you want to integrate to.
RSSI sensitivity RSSI values are used to calculate distance to a reader. These values differ between different phone manufacturers, models and even individual phones. On a device where e.g. the proximity mode works poorly and the user would have to almost touch the reader with his phone, the signal can be boosted by specifying RssiSensitivity.HIGH, and if the device opens a lock at too long range, you would use RssiSensitivity.LOW.
To use the RSSI values received from the lock, without modifications, specify RssiSensitivity.NORMAL.
This could be presented as a setting for the user, or tuned as part of some kind of automatic process.
Scan mode (optional) Mode to indicate to the SDK if the application wants to prioritize power consumption or performance when scanning for BLE readers. The scan mode setting only works in Android 5.0, and will be ignored on 4.3 and 4.4 devices. The default setting is to prioritize the power consumption.
BLE opening triggers List of opening trigger implementations that the application wants to enable. The SDK provides three implementations: Twist and Go, Proximity and Seamless.

To change configuration parameters once scanning is started, update the OrigoScanConfiguration instance with new configuration parameters.

Opening modes⚓︎

There are several types of opening modes supported by a Bluetooth reader. A reader can be configured to support one or more of these opening modes. Support for more opening modes is planned for future releases of the SDK. The modes currently supported are:

  • Proximity aka Tap: The mobile device functions like an access card or tag. This mode is typically used when the mobile device is in close proximity to the reader.
  • Seamless: Opens the lock as the user approaches. This works best with Android 5.0, as the user is not required to power on the screen. It could also be used with older models with the restriction that the user is required to power on the screen. Seamless is not suitable for all readers and environments, consult with your reader provider for more details.
  • Twist and Go: The user rotates the mobile device 90 degrees in one direction and then back again. This opens the closest reader within range.

To implement custom opening mode look at the documentation on (com.hid.origo.api.ble.OrigoOpeningTrigger). The OrigoReaderConnectionController also supports manual opening, however the underlying mechanism still relies on opening triggers.

Reader connection callbacks⚓︎

Callbacks are delivered to the implementing application using the com.hid.origo.api.ble.OrigoReaderConnectionCallback interface. The Bluetooth LE service notifies the connection listener when a reader connection is opened, closed, or has failed. These methods are typically used to give the user feedback using sound, vibration, or the UI. Please note that in some failure cases the Reader parameter in the callback may be null, for example, when the user has performed an open motion with no readers within range. For more information on how to register for and handle callbacks, refer to the javadoc for the com.hid.origo.api.ble.OrigoReaderConnectionCallback and related classes.

Limitations⚓︎

A Bluetooth LE reader can only make one connection at a time. Therefore, when two apps attempt to open simultaneously, using Bluetooth, the app opening can be very slow.

Host-based Card Emulation (NFC)⚓︎

Android version 4.4 introduced a mechanism for an Android device to emulate a contactless card using NFC. This feature is referred to as Host-based Card Emulation (HCE). Detailed information of this feature can be found in the Android API guides.

The SDK contains a service implementing HCE. The service is disabled by default but can be either enabled or disabled using com.hid.origo.api.OrigoReaderConnectionController.

To configure the HCE service you need to be assigned an AID (Application Identifier) by HID Global. When you have received the AID you need to create a seos_hce_service.xml file, add your HCE AID, and provide it as a resource. HCE configuration file res/xml/seos_hce_service.xml:

Limitations⚓︎

The OS selects a HCE service based on the AID. Therefore, if the device has several applications implementing the same AID, the user is either prompted to select which application to use or the system chooses the default application. To avoid the prompt, make sure to register com.assaabloy.mobilekeys.api.hce.HceService as preferred service.

For HceService to function in the background ensure that OrigoMobileKeysApi is initialized. This can be achieved by extending the Android application and initializing OrigoMobileKeysApi in the onCreate() callback.

Device eligibility⚓︎

The Origo SDK contains functionality to check eligibility of the device the SDK is running on. The com.hid.origo.api.OrigoDeviceEligibility class contains information about working/non-working features for the device. It also contains recommended RSSI settings for the device. This feature should be used before initializing the API and can be used to customize the user experience, for example, turning off NFC on devices where NFC is known to fail.

The com.assaabloy.mobilekeys.api.util.DeviceStatusclass contains information about the current status of a device relating to BLE and NFC. It can be asked about permissions, system supported features, and other things.
