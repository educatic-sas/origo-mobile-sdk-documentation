Single Sign-On

Support for Single Sign-On is introduced in version 3.2.0 of the HID Origo SDK. This document provides an overview of how to use this new functionality and support it in parallel with an existing implementation.

Overview⚓︎

HID Origo SDK provides a secure and convenient way for users to sign in to their application using their existing credentials. With SSO, users can request badge in Google Wallet without manually entering the issuance token.

At a high level, there are two steps involved in adding a credential to Google Wallet using SSO.

Note

  • For now, SSO is available for Wallet credential only and SSO doesn't support multiple organizations.
Authorization⚓︎
  • App has to call the authorizationRequest(AuthorizationRequestListener listener) method for authorization.
  • App needs to implement AuthorizationRequestListener to receive callback for Authorization success in the method onAuthorizationSuccess()
  • On successful authorization, the SSO session gets saved in OrigoSDK which will be used to create pass.
Create Pass⚓︎
  • Once the authorization is successful, the app can call the requestPass(CreatePassRequestListener requestListener) method which in-turn triggers the createPass API.
  • App needs to implement CreatePassRequestListener to receive callbacks for Create Pass success in the method onCreatePassSuccess(response: String)
  • onCreatePassSuccess response contains the issuance token which can be used to call endpointSetup(OrigoEndPointSetupCallback origoEndPointSetupCallback,Target target,....) method which will initiate the wallet provisioning flow.

Note

Supported Browsers⚓︎
  • The supported browsers for SSO includes Google Chrome, Microsoft Edge, Firefox and Brave.

SSO Integration⚓︎

Let us look into the step by step process by which users can integrate SSO into their application.

Authorization⚓︎

The AuthorizationRequestListener interface allows app to receive callbacks for authorization success or failure during the Single Sign-On (SSO) process. Implementing this interface is essential for handling authentication outcomes within the application.

void authorizationRequest(AuthorizationRequestListener listener);⚓︎

This method triggers the authentication process for Single Sign-On (SSO) authentication. If the user is not already authenticated, it redirects the user to web-view to log in using their credentials. Once the authentication is successful or fails, the provided AuthorizationRequestListener receives callbacks to handle the corresponding authorization outcome.

Callback methods⚓︎

1. onAuthorizationSuccess()⚓︎

  • Called when the authorization process is successful.
  • App should implement this method to handle post-authentication actions, such as updating the button state
1
2
3
override fun onAuthorizationSuccess() {
        // Handle successful authorization
        }

2. onAuthorizationFailed(response: AuthResponse)⚓︎

  • Called when the authorization process fails.
  • The AuthResponse object carries the appropriate error message from the server, providing insights into the reason for the failure.
  • App should implement this method to handle error scenarios gracefully and provide appropriate feedback to users.
1
2
3
override fun onAuthorizationFailed(response: AuthResponse) {
        // Handle failed authorization
        }

Create Pass⚓︎

The CreatePassRequestListener interface enables app to receive callbacks for create pass success or failure after successful authorization. This is crucial for initiating further actions based on the outcome of the pass creation process.

void requestPass(CreatePassRequestListener requestListener);⚓︎

This method is used to trigger the process of creating a pass (i.e badge) after successful authorization. Once the user is successfully authorized, this method initiates the creation of a pass. Depending on the result of create pass, the provided CreatePassRequestListener receives callbacks to handle the corresponding outcome.

Callback methods⚓︎

1. onCreatePassSuccess(token: String)⚓︎

  • Called when the create pass process is successful.
  • App receive the issuance token, which can be used to start the endpoint setup for initiating the wallet provisioning flow.
1
2
3
override fun onCreatePassSuccess(token: String) {
        // Handle successful pass creation
        }

2. onCreatePassFailed(response: CreatePassResponse)⚓︎

  • Called when the create pass process fails.
  • The CreatePassResponse object contains the appropriate error message for the failure, providing insights into the reason for the failure.
1
2
3
override fun onCreatePassFailed(response: CreatePassResponse) {
        // Handle failed pass creation
        }

getSSOToken() and clearSSOToken()⚓︎

App can make use of the methods getSSOToken() to fetch SSO token and clearSSOToken() to delete SSO token from the storage respectively.

SSO Organization Configuration and Setup⚓︎

Configuration setup phase⚓︎

Step Responsible Party Description
1 Organization Admin and IT Team Provide Redirection/Callback URL to IT team: https://openidservice.webportal.origo.hidglobal.com/origo/openid/{organization-id}/oidc/callback/mobile.
Example URL: https://openidservice.webportal.origo.hidglobal.com/origo/openid/xxxxxxx/oidc/callback/mobile
2 Organization Admin and IT Team Collaborate to set up an application within the Identity Provider (IDP) and obtain the IDP Issuer URL, the IDP Authorization Endpoint, and the IDP Token Endpoint.
3 Organization Admin and IT Team Generate a new application within the IDP's Tenant and acquire the Client ID and Client Secret.

SSO configuration and usage phase⚓︎

Step Responsible Party Description
4 Customer or Partner Service When a Customer or Partner requests wallet activation for their organization, they must provide the IDP configuration.
5 Helix Ticket System Requests related to new SSO configurations or updates to existing configurations will be processed through the Helix ticket system.
6 Organizational Admin Customize the IDP to provide an ID_TOKEN containing the employee's work email in the sub-attribute.
7 IT Team and HID If the IDP has a custom implementation of OIDC, there may be changes required in the HID Origo Passport service to ensure compatibility with the third-party IDP.
8 Organization Admin or Partner Service The organization should have a specified default pass template, and during configuration, the default setting should be set to TRUE.
9 HID SSO has only been validated with OAuth 2.0 and Microsoft Azure IDP. If you use a different IDP, adjustments to the HID Origo backend may be required to ensure support.

User Account Management phase⚓︎

Step Responsible Party Description
10 Organization Admin and IT Team When creating a user in the HID Origo Organization, the user's work email address is required.
11 HID SSO functionality is not supported if the same user is added to multiple organizations.
12 Organization Admin or Partner Service When generating user accounts, it is important to make sure they are set up user's email type to work in order to utilize the Single Sign-On (SSO) feature in HID Mobile Access.
13 Organization Admin or Partner Service User onboarding should be done through the User Management 3.0 API. If user creation is done using the HID Origo Portal, the integrator has to perform additional steps in their Origo integration to use the API and change the user's email type to work in order to utilize the Single Sign-On (SSO) feature in HID Mobile Access.
